### INDEX.md sections produced

The script writes seven sections, each pairing an Obsidian dataview block (fixed literal text) with a static markdown fallback wrapped in `<!-- static-fallback-start: <id> -->` / `<!-- static-fallback-end: <id> -->` markers:

| Section | Dataview | Static fallback content |
|---------|----------|-------------------------|
| Wiki pages | `TABLE … FROM "a4" WHERE kind` | 7-row table: kind, presence, last-updated |
| Stage progress | *(none — static-only)* | 5-row table: Usecase / Arch / Bootstrap / Plan / Impl with per-status counts |
| Open issues | `TABLE … GROUP BY file.folder + " · " + status` | per-type breakdown including review.kind split (finding/gap/question) |
| Drift alerts (N) | `TABLE … FROM "a4/review" WHERE source = "drift-detector"` | per-alert row: review wikilink, target wiki, priority, status |
| Milestones | `TABLE … GROUP BY milestone` | tasks complete / total + open reviews per milestone |
| Recent activity | `TABLE … SORT updated DESC LIMIT 10` | top-10 cross-folder rows |
| Spark (open N) | `LIST FROM "a4/spark" WHERE !status OR status = "open"` | bullet list with flavor (brainstorm/decide) + updated date |

CLI:

```bash
uv run plugins/think/scripts/index_refresh.py <a4-dir>             # write INDEX.md
uv run plugins/think/scripts/index_refresh.py <a4-dir> --dry-run   # render to stdout, no write
uv run plugins/think/scripts/index_refresh.py <a4-dir> --stdout    # write and also print
```

## Design decisions made during the implementation

These shaped the script and skill but were judgment calls not explicit in the ADR. Flag them if next session wants to reconsider.

1. **Dataview blocks AND static fallback, both live in INDEX.md.** Earlier in the session I proposed static-only on the reasoning that the script regenerates accurately enough. Reversed after re-reading the ADR + 0143 handoff: dataview and static serve **complementary** purposes — dataview stays live in Obsidian between compass runs (when users edit issues), static is the frozen snapshot for non-Obsidian viewers (git web, terminal, plain readers). Both belong, even though they will sometimes show different counts.

2. **Stage progress is static-only.** Of the seven sections, only this one has no dataview block. Reason: it mixes wiki-page presence checks with cross-folder issue aggregates; expressing both in a single dataview query is awkward. Documented inline in the SKILL.md and the ADR annotation. The script comment notes this explicitly so future readers don't try to "fix" the omission.

3. **Static fallback wrapped in `<!-- static-fallback-start: <id> -->` / `<!-- static-fallback-end: <id> -->` markers.** Not strictly necessary today (the script overwrites the entire file every run), but the markers make a future surgical-edit mode possible without redesign — useful if INDEX ever grows to a size where full regeneration is wasteful, or if a hook wants to hand-edit one section.

4. **Stage-row breakdown enumerates all observed statuses, not buckets.** First draft grouped statuses into terminal/in-progress/blocked/other, but the "other" bucket lost information (e.g., 1 pending + 1 failing collapsed to "2 other"). Rewrote to enumerate every status with count, ordered terminal → in-progress → blocked → alphabetical. More verbose, more useful.

5. **Archive section omitted entirely.** Per this session's design discussion, under the new spec-as-wiki+issues model "closed = archived" — `status: done | resolved | dismissed | final` is the lifecycle terminal state and there is no semantic role for a separate archive folder of issues. The old INDEX's Archive section had no analog.

6. **All sections always present, never omitted.** Old INDEX had `Omit the Spark section if no open sparks. Omit the Archive section if `a4/archive/` holds no pipeline files`. The new dashboard keeps every section visible (showing "0" or "*No …*") so the layout is predictable and discoverable.

7. **Compass Step 0.3 reads INDEX.md back to summarize Stage progress + Drift alerts.** Old Step 0.6 showed the Pipeline table inline. New Step 0.3 directs compass to read the just-written INDEX.md and pluck those two sections — they answer "where is the workspace at" faster than the others. This is the only section-aware part of compass; the rest treats INDEX as a black-box file written by the script.

8. **Steps 1–3 of compass left untouched.** They still describe the legacy aggregated-file model (`<slug>.usecase.md`, `<slug>.arch.md`, etc.). Adding a one-paragraph legacy note flagged the inconsistency loudly without doing the full rewrite, which is ADR Next Step 6's scope. Touching them now would be undone by the compass redesign.

9. **`disable-model-invocation: true` on the `/think:a4-index` skill.** Same convention as `/think:a4-drift` and `handoff` — the model should not auto-invoke from description keywords. The skill only fires when the user types `/think:a4-index` explicitly.

10. **Skill name `a4-index` (no `think-` prefix).** Per the ADR's namespacing decision (rejected `prefix-based skill naming`), invocation is `/think:a4-index`. Parallel to `/think:a4-drift`.
