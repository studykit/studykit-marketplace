### Detection rules in the script

Five drift categories, each producing a review item with `target: <wiki-basename>`, `wiki_impact: [<wiki-basename>]`:

| Drift kind | Review kind | Priority | Trigger |
|------------|-------------|----------|---------|
| `close-guard` | gap | high | Resolved review item declares `wiki_impact: [<wiki>]` but `<wiki>.md` has no `## Changes` footnote citing the causing issue (the review's `target`). |
| `missing-wiki-page` | gap | high | A `wiki_impact` entry names a wiki page that does not exist at `a4/` root. |
| `stale-footnote` | finding | medium | Footnote definition's wikilink resolves to a non-existent issue or wiki page. |
| `orphan-marker` | finding | low | Inline `[^N]` marker without a `## Changes` definition. |
| `orphan-definition` | finding | low | `## Changes` defines `[^N]:` but no inline `[^N]` marker exists. |

CLI:

```bash
uv run plugins/think/scripts/drift_detector.py <a4-dir>             # write review items
uv run plugins/think/scripts/drift_detector.py <a4-dir> --dry-run   # detect only
uv run plugins/think/scripts/drift_detector.py <a4-dir> --json      # structured stdout
```

## Design decisions made during the implementation

These shaped the script and skill but were judgment calls not explicit in the ADR. Flag them if next session wants to reconsider.

1. **Fingerprint-based dedup via labels.** Each emitted review item carries `labels: [drift, drift:<kind>, drift-cause:<cause-slug>]` (the third label is omitted when the drift has no causal pointer, e.g., orphan markers). The fingerprint is `(drift_kind, target_wiki, cause_slug)`. The script reads existing `review/*.md` files, extracts the fingerprint from labels, and skips re-emission when an existing item with the same fingerprint is `open | in-progress | dismissed`. **Resolved items do not block re-emission** — if drift is detected after a resolution, that resolution did not actually fix it. **Dismissed items DO block** — the user explicitly said "not real drift, ignore", and we honor that.

2. **Self-contained `compute_next_id` instead of importing `allocate_id.py`.** PEP-723 scripts can't cleanly import each other without sys.path gymnastics. The 10-line duplication is preferable to coupling. Both functions read frontmatter `id:` from the same set of folders (`usecase`, `task`, `review`, `decision`) and compute `max + 1`, so the semantic guarantee is preserved.

3. **YAML output uses pyyaml's `default_flow_style=None` (auto).** Lists render inline when short (`wiki_impact: [domain]`, `labels: [drift, 'drift:close-guard', ...]`). Dates round-trip as quoted strings (`created: '2026-04-24'`) — valid YAML, parseable, but visually different from the unquoted form some hand-written examples use. Decision: leave as-is. Quoting is harmless and downstream parsers don't care.

4. **`disable-model-invocation: true` on the `/think:a4-drift` skill.** The ADR designates it as an "optional manual trigger" — the model should not auto-invoke it from the description's keywords. The skill only fires when the user types `/think:a4-drift` explicitly. Same convention used by the existing `handoff` skill.

5. **Skill name dropped the `think-` prefix.** Per the ADR's rejected-alternatives entry on namespacing, the new convention is bare names (e.g., `handoff`, not `think-handoff`). The new skill is named `a4-drift`; full invocation is `/think:a4-drift`. This breaks the older `think-usecase` / `think-arch` / `think-plan` convention, but those names are grandfathered.

6. **Auto-bootstrap step reorder is a behavior change.** Prior structure: commit → drift placeholder. New structure: drift detection → commit. Items emitted by the detector are now bundled into the bootstrap commit instead of being stranded as uncommitted working-tree changes. If anyone relied on the old order (e.g., a wrapper that committed before drift detection), they'll need to adjust.

7. **Compass Step 3 integration deferred.** The ADR lists drift detector invocation from compass Step 3 as a goal, but compass itself needs redesigning for the new layout (separate Next Steps item). Wiring drift detection into the current compass would be wasted work. The ADR Next Steps annotation explicitly notes this deferral.

## Explicitly rejected / not done this session

- **Adding unit tests for `drift_detector.py`.** Would break convention with `allocate_id.py` and `inject_includes.py`, both untested. If the convention changes, do it for all three at once.
- **Compass Step 3 wiring.** Compass redesign is a separate Next Steps item; touching the current compass would be undone immediately. Deferred to that work.
- **`drift-detector` source string variants** (e.g., `drift-detector-r1`). The reviewer agents use `source: usecase-reviewer-r2` style with a round suffix. Drift detection has no analogous "round" semantic — it's a one-shot scan, not an iterative review. Single bare `source: drift-detector` is the only valid value.
- **Drift detector emitting `kind: question` items.** Only `finding` and `gap` are produced. `question` implies human-judgment ambiguity; everything the detector produces is mechanically derivable (file existence, footnote presence, fingerprint match), so `finding` / `gap` are the only honest categories.
- **Calling drift detection from `think-usecase` / `think-arch` / `think-plan` finalization.** The reviewer agents already emit per-finding review items in those flows; the in-situ nudge already prompts the user about wiki updates after each issue edit. Adding a third path would create duplicates. Drift detection is reserved for bulk-generation skills (auto-bootstrap) and explicit user trigger (`/think:a4-drift`) and eventually compass.
