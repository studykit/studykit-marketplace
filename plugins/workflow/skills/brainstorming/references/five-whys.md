# Five Whys

## What It Is
A root cause analysis technique that drills down from a surface-level problem to its fundamental cause by repeatedly asking "Why?" — typically five times. Each answer becomes the subject of the next "Why?" until the root cause is uncovered.

## When to Use
- A problem keeps recurring despite fixes
- Need to move from symptoms to root causes
- The obvious solution doesn't feel like it addresses the real issue
- Debugging a process, system, or interpersonal breakdown

## When NOT to Use
- Divergent brainstorming where multiple ideas are needed (Five Whys converges to a single root)
- Complex systemic problems with multiple interacting causes (use Fishbone/Ishikawa instead)
- The cause is already well understood and action is needed

## Process Steps
1. **State the problem** — Write a clear, specific problem statement. Avoid vague framing.
2. **Ask "Why?" #1** — Why does this problem occur? Write the answer.
3. **Ask "Why?" #2** — Why does [answer #1] happen? Write the answer.
4. **Ask "Why?" #3** — Why does [answer #2] happen? Write the answer.
5. **Ask "Why?" #4** — Why does [answer #3] happen? Write the answer.
6. **Ask "Why?" #5** — Why does [answer #4] happen? Write the answer.
7. **Identify the root cause** — The final answer should point to a systemic, structural, or process-level cause that, if addressed, would prevent the original problem.
8. **Verify the chain** — Read the chain in reverse: "Therefore..." — each step should logically follow.

Note: Five is a guideline, not a rule. Sometimes you reach the root in 3 "Whys"; sometimes it takes 7.

## Facilitator Prompts
- "Let's start with a specific statement. What exactly is happening?"
- "Why is that the case? What's causing it?"
- "That's a good answer — but let's go deeper. Why does that happen?"
- "We're getting close to something systemic. One more 'Why?'"
- "Let's verify: if we fixed [root cause], would the chain above collapse?"
- "Are there multiple answers at this level? Let's branch and explore both paths."

## Example
**Problem**: Customers are canceling their subscriptions after the first month.

1. **Why?** They don't see enough value to justify the price.
2. **Why?** They haven't used the key features that deliver the most value.
3. **Why?** The onboarding flow doesn't introduce those features until week 3.
4. **Why?** The onboarding was designed around feature complexity, not user value.
5. **Why?** The onboarding was built by engineering without input from customer success data on what drives retention.

**Root cause**: Onboarding design process doesn't incorporate customer success insights about which features drive retention.

**Action**: Redesign onboarding flow around "time to first value" metric, prioritizing features that customer success data shows are most correlated with retention.
